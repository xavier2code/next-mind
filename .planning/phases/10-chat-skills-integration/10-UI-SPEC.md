## Component Inventory

### Attachment Bar (modification to `chat-message.tsx`)

Compact horizontal row of file metadata chips displayed below user message prose content. Per D-08: shows filename + type icon + size, click navigates to file management preview.

**Layout:**
```
+----------------------------------------------------------+
| [User avatar] You                                         |
|                                                            |
| User's message text content...                            |
|                                                            |
| [📎 report.pdf (2.3MB)] [📎 data.xlsx (156KB)]          |  <- attachment bar
+----------------------------------------------------------+
```

**Rendering rules:**
- Only rendered for user messages where `message.attachments` is defined and non-empty
- Positioned after the prose div, inside the `flex-1 min-w-0` container
- Container: `flex flex-wrap gap-1.5 mt-2` (matches ChatInput file chips row spacing)
- Each item: `<Link>` wrapping inline-flex row
- Item styling: `inline-flex items-center gap-1 px-2 py-0.5 rounded bg-zinc-100 dark:bg-zinc-800 text-xs text-zinc-600 dark:text-zinc-400 hover:bg-zinc-200 dark:hover:bg-zinc-700 transition-colors`
- Icon: reuse `getTypeIcon(att.fileType)` from `file-chip.tsx` (`h-3.5 w-3.5`)
- Filename: `truncate max-w-[160px]` with `title={att.filename}` (matches FileChip truncation)
- Size: reuse `formatSize(att.size)` from `file-chip.tsx`
- Link href: `/files?id=${att.id}` (navigates to file management page with file selected)
- AI messages never show attachment bar (D-09)

**Message type extension:**
```typescript
// ChatMessageProps message type extends to include optional attachments
interface AttachmentFile {
  id: string;
  filename: string;
  fileType?: 'document' | 'code' | 'data';
  size: number;
}

// message.attachments: AttachmentFile[] | undefined
```

Source: CONTEXT.md D-08, D-09. RESEARCH.md Pattern 2 code example. Styling matches Phase 7 FileChip visual language (same icon sizes, truncation widths, font sizes).

### FileChip Edit Extension (modification to `file-chip.tsx`)

Adds an "edit" button to FileChip when status is `ready` and extracted content is available. Clicking expands an inline textarea showing the extracted Markdown content for editing (D-05, CHAT-05).

**Edit button:**
- Position: between the file size text and the close (X) button
- Icon: Pencil from lucide-react, `h-3 w-3`
- Styling: `ml-0.5 p-0.5 hover:bg-muted-foreground/10 rounded` (matches existing close/retry button styling)
- Visibility: only shown when `status === 'ready'` and `extractedMarkdown` is available
- Tooltip: `aria-label="编辑 {filename} 内容"`

**Inline editor expansion:**
- When edit button is clicked, the FileChip expands vertically to show a textarea below the chip row
- Container: `mt-1 w-full` (full width of chip container)
- Textarea: `w-full min-h-[120px] max-h-[300px] text-sm font-mono p-2 border border-border rounded-md bg-background resize-y`
- Content: pre-populated with `extractedMarkdown` (or previously edited content if user has edited before)
- Action row below textarea: Save button + Cancel button, right-aligned, `flex gap-1.5 justify-end mt-1`
- Save: `variant="outline" size="sm"` with "保存" text; calls `onEditContent(fileId, editedText)`
- Cancel: `variant="ghost" size="sm"` with "取消" text; collapses editor without saving
- Only one file can be in edit mode at a time (clicking edit on another chip collapses the current editor)

**FileChip props extension:**
```typescript
interface FileChipProps {
  // ... existing props
  extractedMarkdown?: string;  // NEW: available when status === 'ready'
  isEditing?: boolean;         // NEW: controlled by parent
  onEditStart?: (fileId: string) => void;    // NEW: request edit mode
  onEditContent?: (fileId: string, content: string) => void;  // NEW: save edited content
  onEditCancel?: () => void;   // NEW: cancel edit mode
}
```

Source: CONTEXT.md D-05 (inline Markdown editor). RESEARCH.md recommends textarea over rich editor libraries. Styling follows existing FileChip patterns (xs icon sizing, muted hover backgrounds, same button variants).

### useFileUpload Hook Extension (modification to `use-file-upload.ts`)

**New state tracking:**
```typescript
// Each PendingFile gains an optional editedContent field
interface PendingFile {
  // ... existing fields
  editedContent?: string;  // NEW: user-edited Markdown content (overrides extractedMarkdown)
}
```

**New behavior:**
- `onEditContent(fileId, content)` stores the edited content in the corresponding PendingFile's `editedContent` field
- `getEditedContent(fileId: string)` returns `editedContent` if set, otherwise `undefined`
- `clearFiles()` also clears all `editedContent` fields (existing behavior, no change needed)
- When handleSend fetches file content for injection, it uses `editedContent` if available, falling back to the API-returned `extractedMarkdown`

Source: RESEARCH.md Anti-Pattern: "Storing edited content server-side for CHAT-05" -- edited content is ephemeral, client-side only.

## Interaction Patterns

### Content Injection Flow (CHAT-01/02/03/04)

| Trigger | Behavior |
|---------|----------|
| User sends message with fileIds | handleSend fetches all file contents via `Promise.all` of `GET /api/files/:id` |
| File status is `ready` with extractedMarkdown | Content block appended to message using D-03 delimiter format |
| File status is not `ready` | File skipped; warning shown to user |
| Total content exceeds 10000 chars | Content truncated at threshold; truncation warning appended to message |
| Edited content exists for a file | Uses `editedContent` instead of `extractedMarkdown` |
| No files or all files skipped | Plain message sent without file content blocks |

**Truncation behavior (D-02, CHAT-04):**
- Threshold: 10000 characters total across all file content blocks
- Truncation happens at file block boundaries (never mid-file delimiter)
- When threshold is reached mid-file: truncate the current file's content, append `[Content truncated...]` marker
- After truncation, remaining files are skipped entirely
- Warning text "文件内容过长，已截断显示前 {N} 字符" appended to the enriched message
- N = actual number of characters included (not the threshold)

**File readiness check (RESEARCH.md Pitfall 1):**
- Before sending, check each file's status from the fetched API response
- Files with `status !== 'ready'` or empty `extractedMarkdown` are skipped
- If any files are skipped, show inline warning: "部分文件仍在处理中，将不会包含在消息中"
- If ALL files are skipped, show error: "所有文件仍在处理中，请稍后再试" and do NOT send the message

Source: CONTEXT.md D-01, D-02, D-03, D-04. RESEARCH.md Pattern 1, Pitfall 1, Pitfall 2. Character threshold of 10000 is Claude's discretion per RESEARCH.md recommendation.

### Inline Edit Flow (CHAT-05)

| Trigger | Behavior |
|---------|----------|
| User clicks edit button on FileChip | Chip expands to show inline textarea with extractedMarkdown |
| User edits content in textarea | Local state updated on each change |
| User clicks "保存" | `editedContent` saved to useFileUpload state, editor collapses, FileChip shows subtle indicator (e.g. Pencil icon becomes blue or a small "edited" badge) |
| User clicks "取消" | Editor collapses, no changes saved |
| User clicks edit on different FileChip | Current editor collapses (discard unsaved changes), new editor opens |
| User sends message | handleSend checks for `editedContent` per file; uses it instead of `extractedMarkdown` |
| Files cleared after send | All `editedContent` cleared (ephemeral by design) |

**Edited indicator on FileChip:**
- When a file has unsaved edits in the textarea: no special indicator (user is actively editing)
- When edits are saved (user clicked "保存"): Pencil icon changes to `text-blue-500` to indicate content has been modified
- This indicator is reset when files are cleared after sending

Source: CONTEXT.md D-05. RESEARCH.md Anti-Pattern on ephemeral editing.

### Attachment Bar Interaction (D-08)

| Trigger | Behavior |
|---------|----------|
| User clicks an attachment item | Navigates to `/files?id={fileId}` (file management page with that file selected for preview) |
| Hover over attachment item | Background darkens (`bg-zinc-200`) to indicate clickability |
| No hover state on mobile | Touch targets remain the full item area (minimum 44px height) |

Source: CONTEXT.md D-08.

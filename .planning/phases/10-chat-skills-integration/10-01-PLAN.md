  <behavior>
    CHAT-01: Given a list of fileIds and a user message, fetch each file via GET /api/files/:id, collect files with status==='ready' and non-empty extractedMarkdown, format them as delimited blocks appended to the message, and return the enriched content plus attachment metadata.

    CHAT-02: Each file block format is exactly: `\n\n---\n📎 {filename} ({fileType}, {size})\n{extractedMarkdown}\n---\n`. The delimiter must use the actual filename, fileType (or 'unknown' if null), and human-readable size via formatSize.

    CHAT-03: Multiple files are concatenated in the order they appear in the fileIds array. Each file gets its own delimited block.

    CHAT-04: Total character count across all file blocks (excluding the user's original message) must not exceed MAX_TOTAL_CHARS (10000). When threshold is reached mid-file, truncate the current file's content at the boundary, append `\n\n[Content truncated...]`, and skip remaining files. Append warning text to enrichedContent: `\n\n文件内容过长，已截断显示前 {N} 字符` where N is the actual character count included.

    Edge case: If some files are not ready (status !== 'ready' or empty extractedMarkdown), skip them. Return a warning string in the result: "部分文件仍在处理中，将不会包含在消息中".

    Edge case: If ALL files are skipped, return the original message unchanged with no file blocks and the warning "所有文件仍在处理中，请稍后再试".

    Edited content support: Accept an optional Map<string, string> of fileId -> editedContent. If present for a file, use editedContent instead of extractedMarkdown.

    Cases:
    - No fileIds -> returns original message, empty attachments, no warning
    - 1 ready file -> returns message + 1 block, 1 attachment, no warning
    - 3 files, 1 not ready -> returns message + 2 blocks, 2 attachments, "部分文件" warning
    - 3 files, all not ready -> returns original message, empty attachments, "所有文件" warning
    - Files exceeding 10000 chars -> truncated with warning
    - Edited content provided -> uses editedContent instead of extractedMarkdown
  </behavior>

## Copywriting Contract

| Element | Copy |
|---------|------|
| Extraction in progress | "Processing..." (displayed in chip where file size was shown) |
| Extraction complete | File size displayed (e.g. "2.3MB") -- same as uploaded state |
| Extraction failed | "Extraction failed" (displayed in chip) |
| Extraction timeout | "Processing timed out" (displayed in chip as error) |
| Extraction partial success | File size displayed + warning in errorMessage (e.g. "First 1000 rows extracted") |

Not applicable. Phase 8 has no standalone pages or lists. Extraction status only appears within existing file chips.

| Error | Copy | User Action |
|-------|------|-------------|

| Extraction timed out | "Processing timed out" | Same as above |

None in this phase. File deletion and retry are deferred to Phase 9 (MGMT-02).

## Component Inventory

### FileChip Status Extension (`src/components/files/file-chip.tsx`)

Phase 7 FileChip supports: `pending`, `uploading`, `uploaded`, `error`.
Phase 8 extends the `uploaded` state lifecycle to include extraction sub-states.

**New status values added to FileChip:**

| Status | Visual |
|--------|--------|
| `processing` | Loader2 spinner (`h-3.5 w-3.5 animate-spin text-blue-500`), "Processing..." text replacing file size, muted background, no close button (extraction is non-cancellable) |
| `ready` | CheckCircle2 icon (`h-3.5 w-3.5 text-emerald-500`), file size text restored, emerald-tinted text (`text-emerald-700`), close button visible |
| `failed` (extraction) | AlertCircle icon (`h-3.5 w-3.5 text-destructive`), error text replacing file size, destructive border (`border border-destructive`), no auto-fade (chip stays visible -- user may reference file or retry later) |

**Status flow within chip:**

```
pending -> uploading -> uploaded -> processing -> ready
                                        \-> failed
```

**Key behavioral change from Phase 7:**
- Phase 7: `uploaded` is the terminal state (chip shows file size, emerald text)
- Phase 8: `uploaded` transitions to `processing` within ~1 second (polling starts immediately after upload response). The `uploaded` state is now transient.
- The `ready` state replaces `uploaded` as the visual terminal state for a successful upload+extraction.

**Close button behavior:**
- `pending`, `uploading`: close removes chip (same as Phase 7)
- `processing`: close button hidden (non-cancellable extraction)
- `ready`: close button visible, removes chip from message
- `failed` (extraction): close button visible, removes chip from message. No auto-fade -- stays until user dismisses or page navigation.

### useFileExtractionStatus Hook (`src/hooks/use-file-extraction-status.ts`)

Polls extraction status for a list of file IDs.

**Interface:**
```typescript
interface FileExtractionStatus {
  [fileId: string]: 'processing' | 'ready' | 'failed';
}

interface UseFileExtractionStatusReturn {
  statuses: FileExtractionStatus;
  isPolling: boolean;
  startPolling: (fileIds: string[]) => void;
  stopPolling: () => void;
}
```

**Behavior:**
- Polls GET /api/files/:id/status every 2 seconds (D-03 discretion: 2s interval)
- When a file transitions to `ready` or `failed`, stop polling that individual file
- When all files reach terminal state (`ready` or `failed`), set `isPolling = false`
- Uses `setInterval` with cleanup on unmount
- On network error during polling: retry next interval silently (no error shown to user -- extraction status is non-critical UX)

### useFileUpload Hook Extension (`src/hooks/use-file-upload.ts`)

**New behavior after upload completes:**
- When a file transitions to `uploaded` status (upload API returns success), immediately start extraction status polling via `useFileExtractionStatus`
- Update the local file state's `status` field to reflect extraction progress: `uploaded` -> `processing` -> `ready` / `failed`
- The `uploadedFile` metadata already stored from Phase 7 (fileId, fileType, etc.) is used to start polling

**Updated file status type:**
```typescript
type FileChipStatus = 'pending' | 'uploading' | 'uploaded' | 'processing' | 'ready' | 'error';
```

## Interaction Patterns

### Extraction Status Polling

| Trigger | Behavior |
|---------|----------|
| Upload API returns success (fileId received) | Immediately start polling GET /api/files/:id/status |
| Poll returns `processing` | Update chip to show spinner + "Processing..." |
| Poll returns `ready` | Update chip to show CheckCircle2 + file size; stop polling this file |
| Poll returns `failed` | Update chip to show AlertCircle + error message; stop polling this file |
| All files terminal | Set `isPolling = false`; no more network requests |

**Polling parameters:**
- Interval: 2000ms (2 seconds) -- Claude's discretion per CONTEXT.md
- Cleanup: Stop polling on component unmount, file removal, or terminal status
- Jitter: None (simple implementation matching project patterns)

### Chip State Transitions

The chip transitions through states with no animation between extraction states (instant swap of icon and text). This matches the existing Phase 7 pattern where state changes are immediate.

```
User uploads file:
  -> chip shows uploading (progress bar, %)
  -> upload completes: chip shows uploaded (briefly, <1s)
  -> polling detects processing: chip shows spinner + "Processing..."
  -> polling detects ready: chip shows checkmark + file size
```

### Non-Cancellable Extraction

Unlike upload errors (which have retry), extraction has no cancel/retry button on the chip. Per CONTEXT.md D-02, retry is from the file management UI (Phase 9). This keeps the chip simple and avoids race conditions.

## API Integration Points

### GET /api/files/:id/status

**Purpose:** Client polls this endpoint to check extraction status.

**Expected response:**
```json
{
  "id": "uuid",
  "status": "processing" | "ready" | "failed",
  "errorMessage": null | "string"
}
```

**Called by:** `useFileExtractionStatus` hook

**No UI component directly calls this API.** All API interaction is encapsulated in the hook.
